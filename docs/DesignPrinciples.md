# Design Principles

---

# Purpose

The purpose of these principles is to guide every future product decision made within the Discogs Intelligence Platform.

Whenever a new feature is proposed, it should be evaluated against these principles.

If a feature conflicts with the principles, it should be redesigned or rejected.

---

# Principle 1

## Automate Research, Not Decisions

DIP exists to reduce the effort required to research a collection.

The platform should never replace the judgement of the collector.

Recommendations should assist decision making, not automate it.

---

# Principle 2

## Transparency Over Complexity

Every recommendation should be understandable.

Users should always know why a record has been highlighted.

Simple explainable logic is preferred over complex black-box algorithms.

---

# Principle 3

## Evidence First

Every insight must be supported by objective evidence.

Examples include:

- Price movement
- Supply
- Demand
- Historical trends
- Market activity

Opinion should never be presented as fact.

---

# Principle 4

## History Is An Asset

Historical information becomes more valuable over time.

The platform should preserve every analysis run.

Nothing should silently overwrite history.

---

# Principle 5

## Intelligence Before Automation

The purpose of DIP is to make collectors smarter.

Not lazier.

The platform should prioritise understanding over automation.

---

# Principle 6

## Collector First

Every feature should answer one question.

"Does this genuinely help collectors?"

If not, it probably doesn't belong.

---

# Principle 7

## Respect User Ownership

Users own:

- their data
- their collection
- their decisions

DIP should never make those decisions for them.

---

# Principle 8

## Explain Every Score

Every score should be reproducible.

Users should always understand how a score was calculated.

---

# Principle 9

## Build For The Long Term

Avoid features that create technical debt.

Prefer sustainable architecture over quick wins.

---

# Principle 10

## Every Feature Must Earn Its Place

Before building a feature ask:

- Does it solve a real problem?
- Does it save time?
- Does it reveal something useful?
- Would collectors miss it if it disappeared?

If the answer is no...

Don't build it.

---

# Principle 11

## Make The Product Enjoyable

DIP should not simply provide data.

It should make collecting more enjoyable.

Opening DIP each week should feel exciting.

---

# Principle 12

## Continuous Improvement

Every release should improve either:

- insight
- usability
- performance
- reliability
- transparency

Never add features simply for the sake of increasing feature count.

---

# Guiding Statement

> Great software doesn't overwhelm users with information.

> It helps them notice what matters.
